# SA Knowledge System — Vault Rules

> This is a vault template. Fork it, rename it for your sub-team (e.g. `NXT-Alpha`, `NXT-Beta`), and start writing your team's specs in `Projects/`.

## What this template provides

- 15 AI agents in `.claude/agents/` — spec orchestrator + 7 specialists + 2 memory + 3 supporting + 2 meta (skill-to-agent, doc-to-vault)
- 13 skills in `.claude/skills/` — PDF, DOCX, XLSX, PPTX, doc-coauthoring, frontend-design, brand-guidelines, skill-creator + 5 Obsidian-native (from kepano)
- Context7 MCP wired via `.mcp.json` — no hallucinated APIs when agents reference frameworks
- Memory system (`Memory/` + `Projects/_meta/architecture-decisions.md`) — ADR log + session logs + rolling summary
- Templates for ADR, SOP, product overview, session log, spec
- SOPs for onboarding, syncing, daily workflow

## Read first when starting a session (in order)

1. `Memory/summary.md` — recent context across sessions (when populated)
2. `Projects/_meta/architecture-decisions.md` — Quick Status table + ADRs
3. `.index/master-index.md` — overview of every note (after `indexer` runs)

Or just run:

```text
Use the session-logger agent: read at session start
```

## Multi-repo architecture

This template is meant to be forked per sub-team. The intended topology:

- `sa-knowledge-system` (this repo) — upstream shared core
- `<TEAM>-<SUB>` — per sub-team forks (e.g. `ICE-Gold`, `NXT-Alpha`)
- Each sub-team owns its `Projects/<PRODUCT>/`, `Memory/`, and `Projects/_meta/`
- Shared core (`.claude/agents/`, `.claude/skills/`, `ProgramType_Skills/`) auto-syncs from upstream daily — see `docs/auto-sync.md`
- Team-owned (`Projects/`, `Memory/`, `reference_data/`, `Tech/SOP/`, `Templates/`, `MOC/`) — never touched by sync

To sync template updates into a sub-team vault: see `Tech/SOP/sync-from-template.md`

## Conventions

- All notes are Markdown with YAML frontmatter
- Link between notes with `[[wikilinks]]` (not relative paths)
- Tag format: `#category/subcategory`
- Every Project note links to its product/team MOC
- Every SOP starts with a clear "When to use this" section

## How to search this vault

1. First: Read `.index/master-index.md`
2. Also useful: `.index/by-tag.md`, `.index/by-type.md`
3. Then: Use Glob (filename pattern) + Grep (content)
4. Check `MOC/` for human-curated topic indexes
5. Cite filenames in answers
6. If `.index/` looks stale → suggest running `indexer` agent

## Answering questions from KB

- ALWAYS search vault before searching web
- ALWAYS cite source note (e.g. "From [[Projects/<product>/overview]]...")
- If vault has no answer, say so explicitly before going to web
- Never make up information not in the vault

## When writing new notes (SA Lead only)

- Pick a template from `Templates/` first
- Add proper frontmatter (title, date, tags)
- Always include `agent_used: <agent-name>` if generated by an agent (audit trail)
- Link to existing related notes with `[[wikilinks]]`
- Update relevant MOC after creating
- After creating/editing notes, suggest running `indexer` agent

## Agent Attribution Convention

Every file generated/saved by an agent MUST include in frontmatter:

- `agent_used: <agent-name>` — which agent created/saved this
- For auto-generated: `generated_by: <agent>`
- For maintained files: `maintained_by: <agent>`

## Agents (15 total)

### Spec workflow (8)

Orchestrator:

- `spec-writer` — Top-level router for SA spec work

7 Specialists:

- `spec-ui-designer` (wraps `sa-designweb`)
- `spec-backend-service` (wraps `spec-service`)
- `spec-api-designer` (wraps `sa-api-design`)
- `spec-report-designer` (wraps `spec_report`)
- `spec-tester` (wraps `test-service`)
- `spec-reviewer` (wraps `review-service`)
- `gateway-thirdparty-api` (wraps `gateway-thirdparty-api`)

### Memory (2)

- `decision-keeper` — Maintains ADR log
- `session-logger` — Maintains session log

### Supporting (3)

- `kb-assistant` — Q&A
- `indexer` — Refresh index
- `db-schema-documenter` — Generate schema reference docs from `.sql` dumps

### Meta (2)

- `skill-to-agent` — Convert your personal skill / prompt template into a proper team agent (asks back until all fields are filled)
- `doc-to-vault` — Import existing documents (PDF, DOCX, XLSX, PPTX, MD, or whole folders) into the vault as proper Markdown notes with frontmatter, wikilinks, and MOC entries (asks back until placement is decided)

## When in doubt

- Ask the user for clarification rather than guessing
- Show what you found before writing/editing
- Read `Memory/summary.md` + `Projects/_meta/architecture-decisions.md` if context unclear
- For spec work — always start with `spec-writer` (it routes for you)
